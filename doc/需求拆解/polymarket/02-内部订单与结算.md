# Polymarket 内部订单与结算

## 范围

- 第一版只做平台内部订单，不向 Polymarket CLOB 提交真实订单。
- 用户用 AFI 下单，后端按下单时 AFI/USDT 价格折算等值 USDT，再按 Polymarket outcomePrices 计算份额。
- 下单后不可卖出，只等待 Polymarket 市场 resolved 后结算。
- 结算从订单轮询升级为市场级聚合结算，同一个 `marketSlug` 只请求一次 Polymarket，然后批量处理该市场下的订单。
- 订单保存 `biz_type` 区分业务入口：`1加密`、`2体育`、`3Up/Down`。

## 计算

```text
afi_usdt_amount = share_amount * outcome_price
afi_amount = afi_usdt_amount / afi_price
max_payout_usdt = share_amount
```

猜中时保留 `payout_usdt_amount = share_amount` 作为中奖USDT等值；结算时读取实时 AFI/USDT 价格，按 `payout_afi_amount = payout_usdt_amount / payout_afi_price` 发放等值 AFI 到用户 AFI 钱包 `valid_num2`。猜错兑付 0。

## 外扣手续费

Polymarket 内部下单手续费使用外扣模式：用户购买份额本身的成本不变，手续费额外从 AFI 钱包扣除，不影响 `share_amount` 和 `max_payout_usdt`。

系统参数：

- `biz_polymarket_trade_fee_ratio`：Polymarket 内部交易手续费比例，单位 `%`。例如配置 `1` 表示 `1%`。

节点减免：

- 从 `t_node_package_order` 查询当前用户已支付成功订单：`status = 1`。
- 多笔节点订单时，取 `pred_order_fee_relief_rate` 最大值作为手续费减免比例。

计算公式：

```text
order_usdt_amount = share_amount * outcome_price
order_afi_amount = order_usdt_amount / afi_price

base_fee_ratio = t_sys_para.biz_polymarket_trade_fee_ratio
relief_ratio = max(t_node_package_order.pred_order_fee_relief_rate)
actual_fee_ratio = max(base_fee_ratio - base_fee_ratio * relief_ratio / 100, 0)

fee_afi_amount = order_afi_amount * actual_fee_ratio / 100
total_pay_afi_amount = order_afi_amount + fee_afi_amount
```

字段含义：

- `afi_amount`：购买份额成本 AFI，不含手续费。
- `fee_ratio`：系统基础手续费比例快照，单位 `%`。
- `fee_relief_ratio`：节点订单手续费减免比例快照，单位 `%`。
- `actual_fee_ratio`：减免后的实际手续费比例快照，单位 `%`。
- `fee_afi_amount`：本单额外扣减的手续费 AFI。
- `total_pay_afi_amount`：实际总扣款 AFI，等于 `afi_amount + fee_afi_amount`。
- `t_polymarket_market.total_fee_afi_amount`：同一个市场累计收取的手续费 AFI。

示例：

```text
用户购买 100 份 Yes
outcome_price = 0.1
afi_price = 0.5
base_fee_ratio = 1%

order_usdt_amount = 100 * 0.1 = 10 USDT
order_afi_amount = 10 / 0.5 = 20 AFI

无节点减免：
fee_afi_amount = 20 * 1% = 0.2 AFI
total_pay_afi_amount = 20.2 AFI

节点减免 80%：
actual_fee_ratio = 1% - 1% * 80% = 0.2%
fee_afi_amount = 20 * 0.2% = 0.04 AFI
total_pay_afi_amount = 20.04 AFI
```

## 下单请求

报价和下单请求由前端显式传入业务类型，后端只做校验和落库，不靠普通市场 `marketSlug` 猜测加密或体育。

```json
{
  "marketSlug": "will-bitcoin-hit-150k-by-june-30-2026",
  "outcomeIndex": 0,
  "shareAmount": 100,
  "bizType": 1
}
```

`bizType` 取值：

- `1` 加密。
- `2` 体育。
- `3` Up/Down。

Up/Down 短周期市场的 `marketSlug` 包含 `-updown-5m-`，如果前端传入的 `bizType` 不是 `3`，后端按参数异常拒绝，避免入口传错导致后台统计不准。

## 市场聚合

- 新增 `t_polymarket_market`，按 `market_slug` 维护市场状态和总金额。
- 下单成功后，同事务累加市场聚合：
  - `order_count`
  - `total_afi_amount`
  - `total_usdt_amount`
  - `total_share_amount`
- 第一版不统计 Yes/No 或每个 outcome 的分别金额，避免多选项市场不兼容。
- 市场状态：
  - `0` 待结算
  - `1` 结算中/消费者待处理
  - `2` 结算完成
  - `3` 待人工复核

订单状态：

- `0` 待结算
- `1` 已猜中
- `2` 未猜中
- `3` 待人工复核
- `4` 已作废/已退款

市场表和订单表都有 `status=1`，但含义不同：市场表的 `1` 表示该市场已经被派发到消费者等待处理；订单表的 `1` 表示用户该笔订单已猜中。

## 核心标识说明

Polymarket 对接里要区分“市场是谁”和“用户买了哪个结果”。当前不单独新增 `question_id` 字段，问题/市场身份由 `market_slug`、`market_id`、`condition_id` 共同覆盖。

市场级标识：

- `market_slug`：当前内部主关联键，用于查市场详情、下单、市场聚合、订单批量结算。
- `market_id`：Polymarket/Gamma 返回的市场 ID 快照，用于保留外部市场编号。
- `condition_id`：Polymarket CLOB/链上条件 ID，后续如果接更深的链上或 CLOB 数据会用到。
- `market_question`：展示用问题文案，方便后台排查，不作为唯一键。

结果级标识：

- `outcome_index`：用户选择结果在 `outcomes` 数组里的下标。
- `outcome_name`：用户选择结果文案，例如 `Yes`、`No`，或者比分、候选项等多结果选项。
- `asset_id/token_id`：用户选择结果对应的 Polymarket CLOB 结果 token。订单表保存为 `t_polymarket_order.asset_id`。

开奖结果标识：

- `asset_ids_json`：市场全部结果 token 数组快照，保存到 `t_polymarket_market.asset_ids_json`。
- `outcomes_json`：市场全部结果名称数组快照，保存到 `t_polymarket_market.outcomes_json`。
- `winning_asset_id`：WebSocket 或最终结算识别到的赢家 token，保存到 `t_polymarket_market.winning_asset_id`。
- `result_outcome_index` / `result_outcome_name`：最终确认的赢家下标和赢家名称。

排查时可以按下面关系理解：

```text
market_slug / market_id / condition_id  确认这是哪个市场
outcome_index / outcome_name / asset_id 确认用户买的是哪个结果
winning_asset_id / result_outcome_*     确认开奖结果对应哪个结果
```

## 下单快照补充

- 下单时除了保存 `market_slug`、`outcome_index`、`outcome_name`，还需要保存用户选择结果对应的 `asset_id/token_id`。
- `asset_id` 从 Polymarket 市场详情里的 `clobTokenIds[outcome_index]` 获取。
- 市场表需要同步保存：
  - `asset_ids_json`：Polymarket市场所有结果的 `asset_id/token_id` 数组快照。
  - `outcomes_json`：Polymarket市场所有结果名称数组快照。
  - `winning_asset_id`：后续WebSocket或最终结算返回的赢家 `asset_id`。
- 如果下单时 Polymarket 没有返回 `clobTokenIds`，或者 `clobTokenIds` 长度小于等于 `outcome_index`，则拒绝下单，避免后续无法通过WebSocket开奖结果关联用户选择。
- 这里默认 Polymarket Gamma API 返回的 `clobTokenIds` 和 `outcomes` 顺序一致，即：

```text
outcomes[0]      -> clobTokenIds[0]
outcomes[1]      -> clobTokenIds[1]
outcome_index    -> clobTokenIds[outcome_index]
```

## 结果判断

- 结算只处理 `umaResolutionStatus = resolved` 的市场。
- `outcomePrices` 中唯一价格为 `1` 的 outcome 作为赢家。
- 出现 50/50、多个赢家、非 0/1 价格、接口失败或字段缺失时，市场进入待人工复核，该市场下订单不自动结算。

## 结算架构

- 结算系统分三层：
  - WebSocket：实时接收 Polymarket 开奖事件。
  - Quartz：兜底扫描已到结束时间但还未派发的市场。
- 延迟队列：统一发奖入口，真正处理订单结算和钱包入账。
- WebSocket 和 Quartz 都不能直接发奖，只能负责“发现这个市场应该结算了”。
- WebSocket 或 Quartz 发现市场需要结算后，先抢占市场状态：

```text
t_polymarket_market.status: 0待结算 -> 1结算中
```

- 抢占成功后，统一投递 `marketSlug` 到延迟队列。
- 延迟队列消费者收到 `marketSlug` 后，调用统一结算入口 `processSettlingMarket(marketSlug)`。
- `processSettlingMarket(marketSlug)` 仍需要重新查询 Polymarket Gamma API 确认最终结果，不能只信WebSocket消息。
- 数据库市场状态是最终准绳，延迟队列消息只是触发器。

延迟队列消费者处理流程：

```text
延迟队列消费者收到 marketSlug
        ↓
查询 t_polymarket_market.status = 1 的市场
        ↓
查询 Gamma API 复核开奖结果
        ↓
查询该 marketSlug 下 status = 0 的订单
        ↓
按 outcome_index 判断每笔订单赢/输
        ↓
读取结算时 AFI/USDT 实时价格
        ↓
猜中订单按 USDT等值 / AFI价格 批量兑付 AFI 到 valid_num2
        ↓
批量更新订单开奖结果、兑付金额、结算时间
        ↓
市场 status = 2 结算完成
```

不新增 `is_win`、`win_status` 等赢输字段。订单是否猜中直接用 `t_polymarket_order.status` 表达，开奖结果保存到 `result_outcome_index/result_outcome_name`；`payout_usdt_amount` 表示中奖USDT等值，`payout_afi_price` 表示结算AFI价格快照，`payout_afi_amount` 表示实际发放AFI数量。

推荐流程：

```text
Polymarket WebSocket market_resolved
        ↓
按 slug 或 winning_asset_id 定位 t_polymarket_market
        ↓
抢占 status=0 的市场为 status=1
        ↓
投递 marketSlug 到延迟队列
        ↓
延迟队列消费者调用 processSettlingMarket(marketSlug)
        ↓
重新查 Gamma API 确认结果
        ↓
批量结算该市场下所有待结算订单
```

Quartz兜底流程：

```text
Quartz 扫描 end_time <= now 且 status=0 的市场
        ↓
抢占 status=0 的市场为 status=1
        ↓
投递同一个 marketSlug 延迟队列
        ↓
延迟队列消费者统一发奖
```

当前阶段 WebSocket 和 Quartz 都会在抢占成功后投递 `marketSlug` 到延迟队列；后台市场页仍保留手动处理 `status=1结算中` 市场的能力，方便运维兜底。

官方依据：

- Polymarket Market WebSocket 文档：`https://docs.polymarket.com/cn/market-data/websocket/market-channel`
- Market Channel 订阅使用 `asset_ids`。
- 开启 `custom_feature_enabled=true` 后，可以接收 `market_resolved` 事件。
- `market_resolved` 事件中包含 `slug`、`asset_ids`、`outcomes`、`winning_asset_id`、`winning_outcome` 等字段，可用于定位市场和赢家。

- 后台市场：`/xms/polymarketMarket`

## 定时任务

- 增量SQL会向 `sys_job` 幂等插入 `Polymarket内部市场结算`。
- 调用目标：`xmsTask.settlePolymarketOrders`。
- Cron：`0/10 * * * * ?`，每10秒扫描一次已结束且待结算的市场。
- 并发策略：`concurrent=1`，禁止Quartz并发执行；同时市场表使用 `待结算 -> 结算中` 状态抢占，避免重复派发。
- 当前版本 Quartz 抢占成功后会投递 Redisson 延迟队列，消费者按 `bizType=30` 调用 `processSettlingMarket(marketSlug)`。
- 新增结算中重投递兜底任务：
  - 调用目标：`xmsTask.redispatchStuckPolymarketMarkets`。
  - 作用：扫描 `status=1结算中` 且 `update_time <= 当前时间 - 10分钟` 的市场，重新投递 `marketSlug` 到同一个延迟队列。
  - 该任务不直接发奖、不修改市场状态，只修复“已抢占但队列消息未投递/服务重启/消费者异常”的卡单场景。
  - 10分钟间隔用于避免上一轮消费者仍在处理时被重复投递；即使出现重复消息，`processSettlingMarket` 仍只处理数据库中 `status=1` 的市场。

## 增量SQL设计

本次不重建订单表和市场表，只补字段和索引。

```sql
ALTER TABLE t_polymarket_order
  ADD COLUMN asset_id varchar(100) DEFAULT NULL COMMENT '用户选择结果对应的Polymarket asset_id/token_id快照' AFTER outcome_name;

ALTER TABLE t_polymarket_order
  ADD KEY idx_asset_id (asset_id);

ALTER TABLE t_polymarket_market
  ADD COLUMN asset_ids_json longtext COMMENT 'Polymarket市场所有结果asset_id数组快照' AFTER market_question,
  ADD COLUMN outcomes_json longtext COMMENT 'Polymarket市场所有结果名称数组快照' AFTER asset_ids_json,
  ADD COLUMN winning_asset_id varchar(100) DEFAULT NULL COMMENT 'WebSocket或结算结果返回的赢家asset_id' AFTER result_outcome_name;

ALTER TABLE t_polymarket_market
  ADD KEY idx_winning_asset_id (winning_asset_id);
```

如果后续正式执行增量SQL，需要先检查线上表是否已经存在同名字段或索引；开发库可以直接执行，生产环境建议包装成幂等脚本。

## 后续代码改造点

- `PolymarketOrderAppServiceImpl.buildMarketPriceSnapshot(...)`：
  - 解析 `outcomes`、`outcomePrices` 的同时解析 `clobTokenIds`。
  - 通过 `clobTokenIds[outcome_index]` 获取用户选择的 `asset_id`。
  - `clobTokenIds` 缺失或长度不足时，直接拒绝报价和下单。
- 创建订单：
  - 写入 `asset_id`。
  - 下单返回对象可带出 `asset_id`，方便前端调试。
- 创建或累加市场聚合：
  - 写入 `asset_ids_json`。
  - 写入 `outcomes_json`。
- 延迟队列：
  - 下单成功后暂时不真实调用 `redissonDelayHandler.add(...)`。
  - 保留方法和中文注释，写明后续接入队列后在事务提交后投递 `marketSlug`。
- 结算：
  - 仍以 `processSettlingMarket(marketSlug)` 作为唯一真实发奖入口。
  - 未来 WebSocket 消费端和 Quartz 派发端都只负责抢占市场状态并投递队列。

## 补充测试点

- 下单成功后，订单表能看到用户选择的 `asset_id`。
- 市场表能看到完整 `asset_ids_json` 和 `outcomes_json`。
- `outcome_index=0` 时，订单 `asset_id` 等于 `clobTokenIds[0]`；`outcome_index=1` 时等于 `clobTokenIds[1]`。
- Polymarket 返回 `clobTokenIds` 为空或长度不足时，下单失败且不扣AFI。
- Quartz到期扫描会把市场从 `0待结算` 改成 `1结算中`，并投递延迟队列，不直接发奖。
- WebSocket收到 `market_resolved` 后会把市场从 `0待结算` 改成 `1结算中`，并投递延迟队列，不直接发奖。
- `status=1` 且 `update_time` 超过10分钟的市场会被重投递兜底任务重新投递队列。
- 后台手动结算仍可调用现有 `processSettlingMarket(marketSlug)` 完成批量发奖。
