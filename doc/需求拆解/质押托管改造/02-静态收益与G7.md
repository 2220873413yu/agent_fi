# 第二批：静态收益与 G7

## 状态

本文按 `doc/0429 G7增长率计算及举例.pdf` 的最终版核心规则 v2.0 重写。

目标是替换当前 101 托管静态收益任务里的占位收益率 `0.5%`，改为按用户伞下团队 TVL 增长计算每日基础静态收益率。

本次确认的实现口径：

- G7 团队 TVL 只统计用户伞下团队，不包含用户自己的托管订单。
- G7 统计单位使用托管 USDT 金额，不使用套餐积分系数。
- 后台拨付订单参与 G7 团队 TVL 统计。
- 周新增小区业绩和全球分红继续使用套餐积分口径，与 G7 静态收益率分开。
- 用户长期指定收益率 `stake_hosting_static_rate > 0` 时优先使用指定收益率。

## 核心公式

每日静态毛收益：

```text
基础静态毛收益 = 托管订单金额 × 当日基础收益率
```

如果订单存在已生效、未退还的 AFI 加速：

```text
加速后静态毛收益 = 基础静态毛收益 × AFI accelerate_rate
```

服务费、用户到账静态收益、团队奖励继续基于加速后的静态毛收益计算。

## 单日增长率 G

单日增长率按用户伞下子团队 TVL 的日变化计算。

```text
G_day = (今日子团队TVL - 昨日子团队TVL) / max(昨日子团队TVL, 100) × 100%
```

参数：

```text
低基数保护 = 100 U
单日增长率上限 = 200%
```

如果计算出的 `G_day > 200%`，则按 `200%` 记录。

如果昨日子团队 TVL 为 `0`，则该日 `G_day` 按 `0%` 处理，避免从 0 做分母。

负增长允许保留真实负数，不做下限截断。

### 团队 TVL 记录方式

下单或后台拨付托管订单生效时，系统查询买家的所有有效上级，并给这些上级在当天的 G7 日汇总记录累加：

```text
team_new_amount += stake_usdt_amount
```

这里不累加买家自己的记录，因为 G7 使用的是团队业绩，不是个人业绩。

101 任务中订单发满完成时，系统查询买家的所有有效上级，并给这些上级在次日的 G7 日汇总记录累加：

```text
team_expired_amount += stake_usdt_amount
```

到期扣减影响次日 G7，不追溯影响当天已发放的静态收益率。

示例：

```text
5月10日，用户B下单 1,000U，B 的上级为 A。
A 在 5月10日的 team_new_amount 增加 1,000U。
B 自己的 G7 记录不增加。

5月20日，B 的订单在 101 任务中发满完成。
A 在 5月21日的 team_expired_amount 增加 1,000U。
5月20日已经发放的收益率不回滚、不重算。
```

## 7 天滚动平均 Gsmooth

`Gsmooth` 是最近最多 7 天 `G_day` 的平均值。

新用户或数据不足 7 天时，有多少天数据就平均多少天。

```text
第1天：Gsmooth = G1
第2天：Gsmooth = (G1 + G2) / 2
第3天：Gsmooth = (G1 + G2 + G3) / 3
...
第7天及以后：Gsmooth = 最近7天 G_day 平均值
```

新用户前几天不强制为 `0%`，增长可以更快反映到静态收益率。

## 日化收益率档位

按 `Gsmooth` 命中日化收益率：

| Gsmooth 范围 | 日化收益率 |
| --- | ---: |
| `G < -20%` | `0.3%` |
| `-20% <= G < -10%` | `0.35%` |
| `-10% <= G < 0%` | `0.4%` |
| `0% <= G < 5%` | `0.5%` |
| `5% <= G < 10%` | `0.6%` |
| `10% <= G < 15%` | `0.75%` |
| `15% <= G < 20%` | `0.9%` |
| `20% <= G < 25%` | `1.1%` |
| `25% <= G < 30%` | `1.3%` |
| `G >= 30%` | `1.5%` |

收益率配置建议后台可维护，默认初始化为上表。

## 未推广用户特殊规则

只拿纯静态的用户，也就是未推广用户：

```text
未回本前：0.5%
回本后：0.2%
```

这里的“回本”按托管订单是否已回本判断。当前订单已有 `is_return_principal` 字段，可作为实现时的订单级判断依据。

如果一个用户有多个托管订单，应按单笔订单分别判断是否回本。

## 例子一：稳定增长

用户个人托管：

```text
10,000 U
```

子团队 TVL：

| 天数 | 子团队 TVL |
| --- | ---: |
| 第1天 | 1,000 U |
| 第2天 | 1,100 U |
| 第3天 | 1,210 U |
| 第4天 | 1,331 U |
| 第5天 | 1,464 U |
| 第6天 | 1,610 U |
| 第7天 | 1,771 U |

单日 G：

```text
第2天 = (1,100 - 1,000) / max(1,000, 100) × 100% = 10.00%
第3天 = (1,210 - 1,100) / 1,100 × 100% ≈ 10.00%
第4天 = (1,331 - 1,210) / 1,210 × 100% ≈ 10.00%
第5天 = (1,464 - 1,331) / 1,331 × 100% ≈ 9.99%
第6天 = (1,610 - 1,464) / 1,464 × 100% ≈ 9.97%
第7天 = (1,771 - 1,610) / 1,610 × 100% ≈ 10.00%
```

`Gsmooth ≈ 10.00%`，命中：

```text
10% <= G < 15%，日化收益率 = 0.75%
```

静态日收益：

```text
10,000 × 0.75% = 75 U
```

## 例子二：低基数爆发保护

用户个人托管：

```text
10,000 U
```

子团队 TVL：

| 天数 | 子团队 TVL |
| --- | ---: |
| 第1天 | 10 U |
| 第2天 | 5,000 U |
| 第3天 | 5,000 U |
| 第4天 | 5,000 U |
| 第5天 | 5,000 U |
| 第6天 | 5,000 U |
| 第7天 | 5,000 U |

单日 G：

```text
第1天：昨日 TVL = 0，G_day = 0%
第2天：(5,000 - 10) / max(10, 100) × 100% = 4,990%
第2天触发单日封顶，G_day = 200%
第3天到第7天：TVL 持平，G_day = 0%
```

按严格公式计算：

```text
第2天 Gsmooth = (0 + 200) / 2 = 100%
第3天 Gsmooth = (0 + 200 + 0) / 3 = 66.67%
第7天 Gsmooth = (0 + 200 + 0 + 0 + 0 + 0 + 0) / 7 = 28.57%
```

第2天和第3天都命中最高档 `1.5%`。第7天 `28.57%` 命中：

```text
25% <= G < 30%，日化收益率 = 1.3%
```

如果业务希望第7天仍按最高档 `1.5%`，需要额外确认是否存在“爆发高值延续”规则；按当前 PDF 的正式公式，代码应使用真实每日 G 列表滚动平均，不应把第3天到第7天重复记为 200%。

实现注意：PDF 例子中出现过 `(0 + 200 + 200) / 3` 的写法，但同一段又说明第3天到第7天新增为 0，因此这里按公式判断应为 `(0 + 200 + 0) / 3`。

## 例子三：爆发后停止推广

场景：

```text
用户个人托管 10,000 U
前期快速增长
第15天后停止推广，子团队 TVL 每天约 -2% 收缩
```

示例走势：

| 天数 | 子团队 TVL | Gsmooth | 正常查表收益率 | 最终收益率 | 日收益 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 第1天 | 1,000 | 0% | 0.5% | 0.5% | 50 U |
| 第2天 | 3,000 | 100% | 1.5% | 1.5% | 150 U |
| 第5天 | 约 6,000 | 40%~85% | 1.5% | 1.5% | 150 U |
| 第10天 | 约 12,000 | 15%~45% | 1.5% | 1.5% | 150 U |
| 第15天 | 12,500 | 约 18% | 0.9% | 0.9% | 90 U |
| 第20天 | 11,800 | 约 8.5% | 0.6% | 0.6% | 60 U |
| 第25天 | 10,900 | 约 2.8% | 0.5% | 0.5% | 50 U |
| 第30天 | 10,200 | 约 0.9% | 0.5% | 0.5% | 50 U |
| 第40天 | 9,200 | 约 -1.5% | 0.4% | 0.4% | 40 U |

解释：

- 停止推广后，收益不会立刻暴跌。
- 随着高增长旧数据滚出最近 7 天窗口，`Gsmooth` 会逐步下降。
- 最终收益率按 `Gsmooth` 命中的收益率区间计算；示例中的天数只是走势参考，不额外触发固定天数惩罚。

如果 `Gsmooth` 继续下降并命中 `G < -20%` 档位，则最终收益率自然降为 `0.3%`。本批不实现“第25天固定降档”之类的额外规则。

## 建议实现方式

### 1. 订单日期字段

建议在托管订单表中增加按天查询字段，用于快速筛选某天新增和某天到期的托管订单。

```text
performance_start_day   业绩开始日期，格式 yyyyMMdd
performance_end_day     业绩结束日期，格式 yyyyMMdd
```

订单生效时写入：

```text
performance_start_day = performance_start_time 对应的 yyyyMMdd
performance_end_day   = performance_end_time 对应的 yyyyMMdd
```

并建立索引：

```text
idx_performance_start_day(performance_start_day)
idx_performance_end_day(performance_end_day)
```

原因：

- G7 计算需要每天知道团队 TVL 的新增和到期变化。
- `performance_start_time`、`performance_end_time` 是 `yyyyMMddHHmmss`，适合精确时间判断。
- `performance_start_day`、`performance_end_day` 是 `yyyyMMdd`，适合定时任务按天快速检索和汇总。

### 2. 用户每日团队 TVL / G 快照

建议新增用户每日团队 TVL / G 快照表，按用户和日期保存一条记录。

这张表不是订单明细表，而是 G7 计算结果表，用来保存每天的团队 TVL、单日 G、滚动 Gsmooth 和最终基础收益率。

```text
user_id
stat_day
previous_team_tvl
current_team_tvl
team_new_amount
team_expired_amount
g_day
g_smooth
base_static_rate
rate_source
calc_status
create_time
```

`current_team_tvl` 使用用户伞下当前有效托管 USDT TVL，不包含用户自己的托管金额。

有效托管 TVL 口径：

```text
产出中托管订单USDT金额
未到期
未删除
支付成功或后台拨付生效
```

后台拨付订单参与 G7 团队 TVL 统计。

每日快照推荐在 101 静态收益任务发放前生成或补齐。

计算方式：

```text
team_new_amount          = 当天伞下生效托管订单USDT金额合计
team_expired_amount      = 当天伞下到期托管订单USDT金额合计
current_team_tvl         = previous_team_tvl + team_new_amount - team_expired_amount
G_day                    = (current_team_tvl - previous_team_tvl) / max(previous_team_tvl, 100) × 100%
G_smooth                 = 最近最多7天 G_day 平均
```

下单时写入或更新当天受影响上级的 `team_new_amount`；订单在 101 中完成时写入次日受影响上级的 `team_expired_amount`。每日快照表保存当天最终计算结果，避免只靠订单表临时计算导致无法追溯历史收益率。

订单表的 `performance_start_day` 和 `performance_end_day` 负责提高扫描效率；每日快照表负责保存当天计算结果。

### 3. 收益率配置表

建议新增 Gsmooth 区间配置表，默认初始化 PDF 的收益率区间。

核心字段：

```text
min_g
max_g
static_rate
sort
status
remark
```

区间采用左闭右开：

```text
min_g <= Gsmooth < max_g
```

最高档 `G >= 30%` 可以使用 `max_g = null` 表示无上限。

### 4. 长期指定用户收益率

长期指定用户收益率不单独建配置表，直接在 `t_user_info` 增加字段。

```text
stake_hosting_static_rate decimal(10,4) not null default 0 comment '托管指定静态收益率，单位%，0表示按G7规则'
```

后台设置规则：

```text
stake_hosting_static_rate >= 0
```

读取规则：

```text
stake_hosting_static_rate > 0   使用该指定收益率
stake_hosting_static_rate = 0   按未推广特殊规则或 Gsmooth 区间收益率计算
```

优先级：

```text
t_user_info.stake_hosting_static_rate > 未推广特殊规则 > Gsmooth 区间收益率
```

示例：

```text
用户A stake_hosting_static_rate = 0
G7 命中收益率 = 0.75%
最终基础收益率 = 0.75%

用户B stake_hosting_static_rate = 1.2
G7 命中收益率 = 0.75%
最终基础收益率 = 1.2%
```

该字段适合长期指定收益率，不记录开始/结束时间，也不保存历史配置变化。

### 5. 接入 101 任务

101 托管静态收益任务执行时：

```text
1. 确定本次收益日 rewardDay。
2. 先生成或读取用户 rewardDay 对应的 G7 收益率快照。
3. 单笔托管订单发放时，读取订单用户的当日基础收益率。
4. 计算基础静态毛收益。
5. 如果订单存在已生效 AFI 加速，乘以 accelerate_rate。
6. 按加速后的静态毛收益计算服务费、用户到账收益和团队奖励。
7. 收益明细或订单发放记录中保存当日使用的基础收益率和最终倍率，方便追溯。
```

当前代码中的 `StakeHostingTaskServiceImpl#calculatePlaceholderStaticRate` 后续应替换为读取 G7 收益率快照。

订单发满完成时：

```text
1. 正常更新订单为已完成。
2. 正常扣减托管业绩。
3. 正常退还已绑定且未退还的 AFI。
4. 给买家的所有有效上级写入次日 team_expired_amount。
5. 标记该订单的 G7 到期统计已处理，避免重复扣减。
```

## 暂不处理

- 本批不新增订单每日收益明细表，仍以订单累计字段、奖励记录和结算明细为主。

## 验收用例

- 稳定 10% 增长命中 `0.75%`，10,000U 订单日收益 75U。
- 低基数从 10U 增至 5,000U，单日 G 被封顶为 200%。
- 最近不足 7 天时按已有天数动态平均。
- 最近超过 7 天时只取最近 7 天滚动平均。
- Gsmooth 命中不同区间时返回对应收益率。
- 未推广未回本订单按 `0.5%`。
- 未推广已回本订单按 `0.2%`。
- AFI 加速只放大静态毛收益，不改变基础收益率快照。
- 服务费、用户到账收益、团队奖励均基于加速后的静态毛收益计算。
