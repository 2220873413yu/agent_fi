# 第五批：团队奖励

## 目标

基于下级静态收益扣除服务费后的净收益，发放直推奖励、团队奖励极差和平级奖励。

本批只覆盖托管团队奖励，不包含节点认购推荐奖励，也不包含每周全球分红的具体结算实现。

## 已知规则

- 团队奖励有效用户定义：读取 `t_user_info.is_valid=1`，该字段表示用户当前持有支付成功且未完成的质押托管订单。
- 有多个托管订单时，只要存在一个未完成订单，下单/订单生命周期维护逻辑应保持 `is_valid=1`；全部达到套餐天数并完成后维护为 `0`。
- `is_valid` 不按 `is_return_principal` 回本状态失效；只有订单运行达到套餐天数、状态变为已完成后，才重新判断用户是否失效。
- 101 任务中如果同一用户多笔订单同批次完成，必须等本轮所有订单状态更新完后，再按用户去重统一刷新 `is_valid`。
- 后台拨付订单每天按规则产生静态收益，计算业绩和升级，但不产生动态奖励。
- 推荐奖励结算到用户 USDT 资产，用户自行提现。
- 直属上级 `is_valid=0/null` 时，直推奖励不入账，并写未到账结算明细用于追溯。
- 极差/平级奖励链路会先过滤 `is_valid=0/null` 的上级，这类上级等同查询不到，不占平级份额。
- 静态收益先扣除服务费，再以扣费后的净收益作为团队奖励基数。
- 不再设置间推奖励。
- 直推奖和极差/平级奖可以叠加：直属上级先拿 10% 直推奖，再按自身实际 F 等级参与极差/平级。
- 无上级时不写平台沉淀结算明细，只在任务日志中记录原因。
- `is_valid=0/null` 的上级，不参与极差/平级奖励链路，不算作已覆盖极差比例，也不占平级拆分份额。
- 后台拨付订单完全静默跳过动态奖励，只发静态收益，不写“后台拨付订单不触发”的动态奖励结算明细。

## 直推奖励

- 直推奖励 = 直推用户静态收益扣除服务费后的净收益 × 10%。
- 自身必须 `is_valid=1` 才可以拿直推奖励。
- 如果自身 `is_valid=0/null`，该笔直推奖励不入账，并写未到账结算明细。
- 后台拨付订单不触发直推奖励，也不写动态奖励跳过明细。

## 团队奖励极差

- 团队奖励极差 = 伞下用户静态收益扣除服务费后的净收益 × 对应等级团队奖励比例。
- 等级团队奖励比例后台可调。
- 极差奖励按“当前等级比例 - 已覆盖等级比例”的差额发放，避免同一段收益被重复按完整比例发放。
- 极差/平级计算前会过滤 `is_valid=0/null` 的用户，这类用户不消耗极差覆盖比例，也不生成极差/平级未到账明细。
- 后台拨付订单不触发团队奖励极差，也不写动态奖励跳过明细。

## 平级奖励

- 平级从 F5 及以上开始。
- 极差奖和平级奖拆开计算：先过滤 `is_valid=0/null` 的上级，再由第一个达到更高等级的有效上级拿完整极差奖。
- 如果该等级为 F5 及以上，则以“第一个到账同级拿到的极差奖金额”作为平级池，再向上穿透低等级寻找后续同级，并按平级公式拆分。
- 第一个到账同级只拿极差奖，不参与本段平级池拆分。
- 平级可穿透低等级：同等级加入本次平级组，低等级跳过且不占份额，遇到更高等级时本次平级组结束。
- 平级池最终按示例固定公式实现：
  - 一个后续同级：`X`。
  - 两个同级：`1/2X`、`1/2X`。
  - 三个同级：`1/2X`、`1/4X`、`1/4X`。
  - 四个同级：`1/2X`、`1/4X`、`1/8X`、`1/8X`。
  - 更多同级时，前序用户依次按 `1/2`、`1/4`、`1/8` 递减，最后一个同级用户承接剩余同份额，确保当前差额池完整分配。

### 极差 + 平级示例

假设：

- 奖励基数 `netReward = 100 USDT`。
- 等级比例：`F3=15%`、`F4=20%`、`F5=25%`。
- 上级链路按真实查询顺序从近到远为：`1004 F3 -> 1003 F4 -> 1002 F5 -> 1001 F5`。

计算：

- `1004 F3` 拿极差：`100 × (15% - 0%) = 15 USDT`。
- `1003 F4` 拿极差：`100 × (20% - 15%) = 5 USDT`。
- `1002 F5` 拿极差：`100 × (25% - 20%) = 5 USDT`，不再拿本段平级奖。
- F5 平级池等于 `1002` 拿到的极差奖 `5 USDT`。
- `1001 F5` 是后续同级，按 1 人平级组拿完整平级池：`5 USDT`。

如果链路为 `F5 -> F5 -> F5 -> F6`：

- 第一个 F5 先按 `F5比例 - 已覆盖比例` 拿完整极差。
- 后续两个 F5 再以第一个 F5 的极差奖作为平级池，按 `1/2、1/2` 拆分平级奖。
- F6 再按 `F6比例 - F5比例` 拿极差补差。

如果链路为 `F5 -> F2 -> F5 -> F3 -> F5 -> F5 -> F6`：

- 第一个 F5 先按 `F5比例 - 已覆盖比例` 拿完整极差。
- F2、F3 都是低等级，会被当前 F5 平级组穿透跳过，不参与平级，也不占份额。
- 后续三个 F5 属于同一个平级组，再以第一个 F5 的极差奖作为平级池，按 `1/2、1/4、1/4` 拆分平级奖。
- 遇到 F6 后，F5 平级组结束；F6 再按 `F6比例 - F5比例` 拿极差补差。

以用户 `1028` 出静态收益为例，若上级链路从近到远为：

`1027 F3 -> 1026 F5 -> 1025 F2 -> 1024 F5 -> 1023 F3 -> 1015 F5 -> 1014 F5 -> 1013 F6`

- `1027 F3` 先按 F3 比例拿极差。
- `1026 F5` 按 `F5比例 - F3比例` 拿极差，不拿本组平级奖。
- `1024、1015、1014` 都属于 F5 后续平级组，按 F5 平级池拆分平级奖。
- `1025 F2、1023 F3` 被穿透跳过，不占平级份额。
- 遇到 `1013 F6` 后，F5 平级组结束，`1013 F6` 按 `F6比例 - F5比例` 拿极差。

如果链路为 `F5 -> F6 -> F5`：

- 第一个 F5 的平级段在遇到 F6 时结束。
- F6 按 `F6比例 - F5比例` 拿极差。
- 后面的 F5 不再参与前面 F5 的平级，也不会倒挂参与 F6 的奖励。

如果链路为 `F3 -> F4 -> F5 -> F5无未出局订单 -> F5 -> F5`：

- 极差/平级奖励链路先过滤掉 `is_valid=0/null` 的 F5。
- 实际参与计算的链路为 `F3 -> F4 -> F5 -> F5 -> F5`；如果中间夹着 F1-F4 低等级，也会被 F5 平级组穿透跳过。
- F5 平级池仍等于第一个有效 F5 的极差奖。
- 第一个有效 F5 只拿极差，后续两个有效 F5 按 `1/2、1/2` 拆分平级池。

## 平级奖首个同级让出一半口径

- 适用范围：F5 及以上平级奖。
- 如果当前等级组没有后续同级，第一个同级仍拿完整极差奖，不产生平级奖。
- 如果当前等级组存在后续同级，第一个同级只拿本段极差池的 50%，剩余 50% 作为平级池给后续同级拆分。
- 后续同级拆分公式保持不变：1 个后续同级拿剩余平级池的 100%；2 个后续同级按 1/2、1/2；3 个后续同级按 1/2、1/4、1/4；4 个后续同级按 1/2、1/4、1/8、1/8。
- 低等级继续穿透且不占份额；遇到更高等级时，当前等级的平级组终止。

示例：净静态收益 100 USDT，F4=20%，F5=25%，F6=30%。

```text
a(F4) -> b(F5) -> d(F6) -> f(F6)
```

```text
a 拿 20 USDT 极差
b 拿 5 USDT 极差
d 的 F6 极差池为 5 USDT，但存在后续同级 f，所以 d 拿 2.5 USDT
f 拿剩余平级池 2.5 USDT
```

多个后续 F6：

```text
a(F4) -> b(F5) -> d(F6) -> f(F6) -> g(F6) -> h(F6)
```

```text
d 拿 2.5 USDT 极差
f/g/h 共享剩余 2.5 USDT，按 1/2、1/4、1/4 拆
f = 1.25 USDT
g = 0.625 USDT
h = 0.625 USDT
```

## 已确认口径

- F 等级使用实际奖励等级：`max(game_level, min_game_level, admin_game_level)`。
- 有效用户统一读取 `t_user_info.is_valid`，该字段由下单生效和托管订单完成流程维护，用来代表是否持有支付成功且未完成的质押托管订单。
- 极差奖励按差额发放。
- 平级奖励按“先发极差，再以极差金额作为平级池”的口径发放。
- 平级奖励从 F5 及以上开始，按“首个同级只拿极差、后续同级加入、低级跳过、高级终止”的穿透低等级口径发放。
- 平台沉淀暂不做单独钱包流水。
- 团队奖和静态收益在同一个托管每日发放任务中处理。
- 后台拨付订单不触发动态奖励时完全静默，不写跳过明细。

## 表结构影响

已新增或复用：

- 等级配置复用 `t_user_level_config.team_reward_ratio`。
- 团队奖励结算明细使用 `t_stake_hosting_reward_settlement`。
- 平级奖励通过结算明细 `reward_type = 4` 区分。

本批不新增平台沉淀钱包流水。

## 后端影响

- 静态收益发放完成后，以扣除服务费后的净收益作为团队奖励基数。
- 发放 USDT 到 `UserMoney.validNum1`。
- 写 `RewardRecord`。
- 后台拨付订单必须排除动态奖励。
- 用户等级升级依赖个人托管和小区托管之和。
- 托管到期订单完成后，用户业绩会扣减，等级计算需要考虑扣减后的业绩。

## 验收用例

- 不再保留间推奖励作为待实现规则。
- 直推奖励比例为 10%。
- 直推奖和极差/平级奖可以叠加。
- 直属上级 `is_valid=0/null` 时，直推奖励写未到账结算明细。
- 极差/平级链路中 `is_valid=0/null` 的上级会被过滤，不占平级份额，不覆盖极差比例。
- 无上级时只打日志，不写平台沉淀结算明细。
- 后台拨付订单不产生动态奖励，也不写动态奖励跳过明细。
- 极差奖励按等级比例正确。
- 平级从 F5 及以上开始。
- 多个同级 F5 的平级分配按确认后的公式正确。
- 同等级之间夹着低等级时，低等级不占平级份额；遇到更高等级时，当前等级平级组结束。
- F1-F9 等级默认值完整保存。
