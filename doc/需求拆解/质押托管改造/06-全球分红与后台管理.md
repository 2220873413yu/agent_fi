# 第六批：全球分红与后台管理

## 目标

实现每周全球分红结算和相关后台管理能力。

## 已知规则

- 全球分红每周日晚上 24 点结算。
- 所有同等级节点，按照每周全球分红小区权重快照差值加权分。
- 后台可调比例。
- 全球分红资金来源于服务费奖池。

## 分红金额来源

全球分红每周可分配金额来自全球分红奖池。

奖池收入包括：

- 101 托管静态收益任务每天按托管周期实际扣除的分红服务费入池金额。
- 后台手动增加的奖池金额。

托管周期对应的服务费比例按套餐配置读取，例如 1天 50%、30天 30%、90天 20%、180天 10%、360天 5%。这里的 `50%` 是 1 天周期服务费比例，不是所有服务费再统一按 50% 入池。

奖池支出包括：

- 每周日全球分红实际发放金额。
- 后台手动扣减的奖池金额。

每周分红任务执行时，需要基于奖池当前余额确定本周可发放金额。实际发放后，需要从奖池余额中扣除，并写奖池流水。

## 每周分红分配口径

每周全球分红按“同等级节点的本期分红权重”加权分配。

全球分红分为两个阶段：

```text
阶段一：下单生效和订单到期实时维护 t_user_info 上的当前全球分红权重。
阶段二：每周 102 分红任务读取当前小区权重，固化本周快照，并与上一期快照做差。
```

`t_user_info` 建议新增并维护以下当前权重字段：

```text
global_dividend_weight              我的全球分红权重
global_dividend_umbrella_weight     团队全球分红权重
global_dividend_community_weight    小区全球分红权重
```

当前权重维护口径：

```text
订单权重 = 订单 performance_points
        或 stake_usdt_amount × performance_coefficient 兜底

下单生效：
1. 给下单人增加 global_dividend_weight。
2. 给所有上级增加 global_dividend_umbrella_weight。
3. 重算受影响上级的 global_dividend_community_weight。

订单到期：
1. 给下单人扣减 global_dividend_weight。
2. 给所有上级扣减 global_dividend_umbrella_weight。
3. 重算受影响上级的 global_dividend_community_weight。

小区权重 = 所有直推区权重合计 - 最大直推区权重
直推区权重 = 直推用户 global_dividend_weight + global_dividend_umbrella_weight
```

每周 102 分红任务生成用户级周快照：

```text
当前小区权重 = t_user_info.global_dividend_community_weight
上一期小区权重 = 上一条全球分红周快照.community_weight
本期分红权重 = max(当前小区权重 - 上一期小区权重, 0)
```

必须保存所有用户本周快照，包括当前小区权重下降、为 0、或本期分红权重为 0 的用户。否则下一周会拿错上一期基准，例如第二周从 10000 降到 8000 时必须保存 8000，第三周才能按 `12000 - 8000 = 4000` 计算。

已确认口径：

- 参与分红的等级使用 F 等级配置。
- 各 F 等级的分红比例使用 `t_user_level_config.global_fee_dividend_ratio`。
- 同一个全球分红奖池先按 F1-F9 配置比例拆分成各等级奖池。
- 每个等级内部，按 `dividend_weight > 0` 的用户加权分配。
- `dividend_weight <= 0` 的用户只保留周快照，不参与同等级分母，也不生成分红明细。
- 如果某等级正数分红权重总和为 0，则该等级本周暂不发放，未发放金额留在奖池。

示例：

```text
本周全球分红可分配金额 = 10,000 USDT
F5 全球手续费分红比例 = 8%
F5 本周等级奖池 = 10,000 × 8% = 800 USDT

F5 用户 A 本期分红权重 = 20,000
F5 用户 B 本期分红权重 = 60,000
F5 本期分红权重总和 = 80,000

A 可得 = 800 × 20,000 / 80,000 = 200 USDT
B 可得 = 800 × 60,000 / 80,000 = 600 USDT
```

非正数示例：

```text
同等级用户 A 本期分红权重 = 0（当前小区权重低于上一期）
同等级用户 B 本期分红权重 = 0
同等级用户 C 本期分红权重 = 4,700
同等级用户 D 本期分红权重 = 3,000

同等级分红总权重 = 4,700 + 3,000 = 7,700
```

A 和 B 不参与分红，也不生成全球分红明细。

## 奖池扣减口径

每周全球分红任务完成发放后，按实际发放总额扣减全球分红奖池。

需要写入：

- 全球分红结算批次。
- 全球分红发放明细。
- 用户 USDT 钱包流水。
- 奖励记录。
- 奖池支出流水。

奖池支出流水需要记录：

- 分红批次号。
- 本次计划分红金额。
- 本次实际发放金额。
- 扣减前奖池余额。
- 扣减后奖池余额。

如果存在未分完金额，本阶段先不确定处理方式，后续业务确认后再定。

## 待确认问题

- 节点等级使用 `node_level` 还是 `game_level`。
- 各等级分红比例是否从同一个奖池拆分。
- 未分完金额如何处理。
- 重算/补发如何防重复。

可能需要新增：

- 全球分红奖池流水表。
- 全球分红配置表。
- 全球分红权重周快照表。
- 全球分红结算批次表。
- 全球分红明细表。

## 后端影响

- 新增每周任务。
- 按等级和本期分红权重分配。
- 每周分红前读取全球分红奖池余额。
- 每周分红后扣减奖池，并写奖池支出流水。
- 发放 USDT 并写奖励记录。
- 后台支持查询、导出、补发、重算。
- 后台支持奖池余额查询和手动调增调减。

## 验收用例

- 每周只结算一次。
- 每周分红金额来自全球分红奖池。
- 同等级用户按本期分红权重分配。
- 后台调比例生效。
- 后台手动增加、扣减奖池后，奖池余额和流水正确。
- 每周发放后，奖池余额按实际发放金额扣减。
- 重复任务不重复发放。
