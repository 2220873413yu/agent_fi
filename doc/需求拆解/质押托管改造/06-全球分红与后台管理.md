# 第六批：全球分红与后台管理

## 目标

实现每周全球分红结算和相关后台管理能力。

## 已知规则

- 全球分红每周日晚上 24 点结算。
- 所有同等级节点，按照每周新增业绩加权分。
- 后台可调比例。
- 全球分红资金来源于服务费奖池。

## 分红金额来源

全球分红每周可分配金额来自全球分红奖池。

奖池收入包括：

- 101 托管静态收益任务每天按托管周期实际扣除的分红服务费入池金额。
- 后台手动增加的奖池金额。

托管周期对应的服务费比例按套餐配置读取，例如 1天 50%、30天 30%、90天 20%、180天 10%、360天 5%。这里的 `50%` 是 1 天周期服务费比例，不是所有服务费再统一按 50% 入池。

奖池支出包括：

- 每周日全球分红实际发放金额。
- 后台手动扣减的奖池金额。

每周分红任务执行时，需要基于奖池当前余额确定本周可发放金额。实际发放后，需要从奖池余额中扣除，并写奖池流水。

## 每周分红分配口径

本阶段先按“同等级节点按每周新增业绩加权分配”记录需求，具体每周新增业绩口径先使用现有小区业绩相关口径，后续再根据业务确认继续细化。

当前临时口径：

- 参与分红的等级使用 F 等级配置。
- 各 F 等级的分红比例使用 `t_user_level_config.global_fee_dividend_ratio`。
- 同一个全球分红奖池先按 F1-F9 配置比例拆分成各等级奖池。
- 每个等级内部，当前先按用户现有小区业绩占该等级现有小区业绩总和的比例加权分配。
- 如果用户现有小区业绩为 0，则该用户本周在对应等级内不参与加权分配。
- 如果某等级现有小区业绩总和为 0，则该等级本周暂不发放，未发放金额留在奖池。
- 后续确认每周新增业绩快照后，再把权重来源从“现有小区业绩”替换为“每周新增小区业绩”。

示例：

```text
本周全球分红可分配金额 = 10,000 USDT
F5 全球手续费分红比例 = 8%
F5 本周等级奖池 = 10,000 × 8% = 800 USDT

F5 用户 A 本周新增小区业绩 = 20,000
F5 用户 B 本周新增小区业绩 = 60,000
F5 本周新增小区业绩总和 = 80,000

A 可得 = 800 × 20,000 / 80,000 = 200 USDT
B 可得 = 800 × 60,000 / 80,000 = 600 USDT
```

这里的“本周新增小区业绩”后续需要再确认：

- 是取本周新增托管订单形成的小区业绩。
- 还是取用户等级计算时已有的小区业绩增量快照。
- 是否只统计用户购买订单，是否排除后台拨付订单。
- 是否按订单创建时间、支付时间、产出开始时间或生效时间归属到本周。

## 奖池扣减口径

每周全球分红任务完成发放后，按实际发放总额扣减全球分红奖池。

需要写入：

- 全球分红结算批次。
- 全球分红发放明细。
- 用户 USDT 钱包流水。
- 奖励记录。
- 奖池支出流水。

奖池支出流水需要记录：

- 分红批次号。
- 本次计划分红金额。
- 本次实际发放金额。
- 扣减前奖池余额。
- 扣减后奖池余额。

如果存在未分完金额，本阶段先不确定处理方式，后续业务确认后再定。

## 待确认问题

- 节点等级使用 `node_level` 还是 `game_level`。
- 每周新增小区业绩统计口径。
- 各等级分红比例是否从同一个奖池拆分。
- 未分完金额如何处理。
- 重算/补发如何防重复。

## 表结构影响

可能需要新增：

- 全球分红奖池余额表。
- 全球分红奖池流水表。
- 全球分红配置表。
- 每周新增小区业绩快照表。
- 全球分红结算批次表。
- 全球分红明细表。

## 后端影响

- 新增每周任务。
- 按等级和每周新增小区业绩权重分配。
- 每周分红前读取全球分红奖池余额。
- 每周分红后扣减奖池，并写奖池支出流水。
- 发放 USDT 并写奖励记录。
- 后台支持查询、导出、补发、重算。
- 后台支持奖池余额查询和手动调增调减。

## 验收用例

- 每周只结算一次。
- 每周分红金额来自全球分红奖池。
- 同等级用户按本周新增小区业绩权重分配。
- 后台调比例生效。
- 后台手动增加、扣减奖池后，奖池余额和流水正确。
- 每周发放后，奖池余额按实际发放金额扣减。
- 重复任务不重复发放。
