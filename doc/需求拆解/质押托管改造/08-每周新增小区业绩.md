# 第八批：每周新增小区业绩

## 状态

本文已按新版口径重写。业务定义以 `10-新增小区业绩定义.md` 为准。

本批目标是为每周全球分红提供可查询、可追溯的“本周新增小区业绩”权重。

## 已确认口径

- 周期按自然周：周一 `00:00:00` 到周日 `23:59:59`。
- 时间字段使用 `yyyyMMddHHmmss` 格式的 long 值。
- 托管订单参与周业绩时不直接使用 USDT 金额，而是使用托管积分。
- 托管积分公式：

```text
托管积分 = 托管金额 × 套餐积分系数
```

- 套餐积分系数在套餐配置中维护。
- 用户下单时，需要把当时的套餐积分系数和托管积分写入订单快照。
- 后续套餐积分系数调整，不影响历史订单。
- 当前全球分红套餐权重：

```text
1天   = ×0
30天  = ×1
90天  = ×2
180天 = ×3
360天 = ×6
```

## 新增小区业绩定义

某个用户 A 的每个直推用户，都是 A 的一条直推区。

某条直推区有效积分：

```text
直推用户本人当前有效托管积分 + 直推用户伞下团队当前有效托管积分
```

用户 A 当前小区有效积分：

```text
所有直推区当前有效积分合计 - 最大直推区当前有效积分
```

本周新增小区业绩：

```text
本周末小区有效积分快照 - 上周末小区有效积分快照
```

因此，本周新增小区业绩允许为负数、0 或正数。

## 自己下单与上级计算

用户自己下单，不计入自己的小区新增业绩。

原因是小区业绩只来自自己的直推区，用户本人不属于自己的任何直推区。

但用户下单会影响所有上级的对应直推区。

例如链路：

```text
A
└── B
    └── C
        └── U
```

U 下单时：

```text
U 自己的小区业绩不因自己下单增加
C 的对应直推区增加 U 的托管积分
B 的对应直推区增加 U 的托管积分
A 的对应直推区增加 U 的托管积分
```

## 到期扣减口径

订单到期后，该订单积分需要从对应直推区有效积分中扣除。

但不能简单按到期积分直接扣所有上级的小区业绩，必须先重算直推区快照。

例如 A 有三条直推区：

```text
甲区 = 5,000
B区 = 2,000
C区 = 1,000
```

此时 A 的小区有效积分：

```text
B区 + C区 = 3,000
```

如果 5 周后甲区到期扣除 1,200：

```text
甲区 = 3,800
B区 = 2,000
C区 = 1,000
```

甲区仍然是最大区，所以 A 的小区有效积分仍然是：

```text
B区 + C区 = 3,000
```

本次到期对 A 的本周新增小区业绩影响为：

```text
3,000 - 3,000 = 0
```

如果到期发生在小区线路，或大区切换导致小区快照下降，则本周新增小区业绩可以为负数。

## 处理流程

订单支付成功或后台拨付生效后：

```text
1. 记录 performance_start_time。
2. 根据套餐天数计算 performance_end_time。
3. 保存 performance_coefficient 和 performance_points 快照。
4. 发送周业绩统计事件。
5. 消费端查询购买用户和所有上级。
6. 对受影响用户重算上周末和本周末直推区有效积分快照。
7. 写入 previous_community_performance、current_community_performance 和 community_new_performance。
```

订单到期完成后：

```text
1. 托管订单正常完成。
2. 扣减用户托管业绩。
3. 将 weekly_expire_performance_status 标记为队列中。
4. 发送到期周业绩重算事件。
5. 消费端按到期所在周重算受影响上级的小区有效积分快照。
```

说明：`weekly_performance_*` 只追踪订单生效时的周新增处理；`weekly_expire_performance_*` 只追踪订单到期完成后的周小区业绩重算。两套状态拆开，方便排查新增和到期是否分别执行。

## 与全球分红关系

每周全球分红使用：

```text
t_stake_hosting_weekly_community_performance.community_new_performance
```

等级奖池先按 `t_user_level_config.global_fee_dividend_ratio` 从全球分红总池中切分：

```text
等级奖池 = 全球分红总池 × global_fee_dividend_ratio / 100
```

同等级用户之间，再按正数周新增小区积分占比分配：

```text
用户分红 = 等级奖池 × 用户正数周新增小区积分 / 同等级正数周新增小区积分合计
```

参与规则：

```text
community_new_performance > 0   参与同等级分红权重
community_new_performance = 0   不参与分红，不生成分红明细
community_new_performance < 0   不参与分红，不生成分红明细
```

负数只表示本周小区有效积分净减少，不作为扣款，也不抵扣同等级正数用户的分红分母。

例如同等级内：

```text
A = -800
B = 0
C = 4,700
D = 3,000
```

则同等级分红总权重：

```text
4,700 + 3,000 = 7,700
```

## 验收用例

- 订单保存套餐积分系数快照，后续套餐系数调整不影响历史订单。
- 1 天订单生成 0 积分；30 天 1000U 订单生成 1000 积分；90 天 500U 订单生成 1000 积分。
- 用户自己下单，不增加自己的小区新增业绩。
- 用户下单后，所有上级对应直推区有效积分变化。
- 到期发生在最大区且最大区仍不变时，小区新增业绩不扣减。
- 到期发生在小区线路时，小区新增业绩允许为负数。
- 后台周业绩页面可查到负数、0、正数。
- 全球分红只使用 `community_new_performance > 0` 的用户。
- `community_new_performance <= 0` 的用户不生成全球分红明细。
