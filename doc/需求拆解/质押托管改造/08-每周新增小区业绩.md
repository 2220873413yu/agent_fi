# 第八批：每周新增小区业绩

## 目标

为每周全球分红提供“本周新增小区业绩”权重。

本需求先明确计算口径和数据落库方式，后续再接入每周全球分红任务。

## 已确认口径

- 推荐关系已有数据来源，不再额外保存 `direct_user_id` 或 `line_user_id`。
- 直推关系使用 `t_user_info.invite_user_id`。
- 上下级闭包关系使用 `t_user_relation`。
- 用户入金后，先形成购买用户自己的本周个人新增业绩。
- 再沿 `t_user_relation` 的上级链路，累加所有上级的本周团队新增业绩。
- 每个用户的本周新增小区业绩，按其所有直推区计算。

## 本周统计周期

每周新增小区业绩按自然周统计。

统计周期固定为：

```text
周一 00:00:00 至 周日 23:59:59
```

周时间字段建议使用 `yyyyMMddHHmmss` 格式的 long 值保存，方便按范围查询和任务过滤。

例如：

```text
week_start_time = 20260504000000
week_end_time   = 20260510235959
```

## 本周有效新增订单口径

每周新增小区业绩只统计“本周新增，并且本周结束时仍未到期”的托管订单。

订单是否计入本周新增业绩，按以下条件判断：

```text
订单生效时间 >= 本周开始时间
订单生效时间 <= 本周结束时间
订单到期时间 > 本周结束时间
```

同时满足以上条件的订单，才进入本周新增业绩统计。

如果订单的到期时间小于等于本周结束时间，说明该订单在本周结算前已经到期完成。为了简化统计逻辑，这类订单不计入本周新增业绩，不初始化周新增业绩数据，也不参与本周新增小区业绩计算。

也就是说，不采用“先初始化新增业绩，再扣除本周到期业绩”的方式，而是在订单事件消费前先过滤。只有本周结束时仍有效的订单，才会进入后续个人新增、团队新增和小区新增计算。

### 有效订单示例

本周统计周期：

```text
2026-05-04 00:00:00 至 2026-05-10 23:59:59
```

用户关系：

```text
A
├─ B
│  └─ U
└─ C
   └─ V
```

本周订单：

```text
订单1：
购买用户：U
金额：5,000 USDT
生效时间：2026-05-05 10:00:00
到期时间：2026-06-04 10:00:00

订单2：
购买用户：V
金额：3,000 USDT
生效时间：2026-05-06 10:00:00
到期时间：2026-05-09 10:00:00
```

订单1的生效时间在本周内，且到期时间晚于本周结束时间，所以订单1计入本周新增业绩。

订单2虽然生效时间也在本周内，但到期时间落在本周周期内，说明本周结算前已经到期完成，所以订单2不计入本周新增业绩，也不初始化周新增业绩数据。

因此，本周有效新增业绩只计算订单1。

对用户 A 来说：

```text
B 区有效新增业绩 = 5,000
C 区有效新增业绩 = 0

A 本周有效新增小区业绩
= B区 + C区 - 最大区
= 5,000 + 0 - 5,000
= 0
```

如果 C 区还有另一笔未到期订单：

```text
订单3：
购买用户：V
金额：2,000 USDT
生效时间：2026-05-08 12:00:00
到期时间：2026-06-07 12:00:00
```

订单3计入本周新增业绩。

则用户 A 的本周有效新增小区业绩为：

```text
B 区有效新增业绩 = 5,000
C 区有效新增业绩 = 2,000

A 本周有效新增小区业绩
= 5,000 + 2,000 - max(5,000, 2,000)
= 7,000 - 5,000
= 2,000
```

### 跳过订单处理说明

周新增业绩消费订单事件时，先判断订单是否满足有效新增条件。

如果订单不满足条件：

```text
不初始化购买用户的周新增业绩记录；
不初始化任何上级用户的周新增业绩记录；
不累加个人新增业绩；
不累加团队新增业绩；
不参与新增小区业绩重算；
只将订单标记为无需统计或已跳过。
```

如果订单满足条件，再进入正常统计流程：

```text
1. 初始化或更新购买用户本周个人新增业绩。
2. 查询购买用户所有上级。
3. 初始化或更新所有上级本周团队新增业绩。
4. 按直推区重新计算受影响用户的本周有效新增小区业绩。
```

## 基础定义

### 本周个人新增业绩

用户本人在本周生效的托管订单金额。

例如用户 A 本周自己购买 1000 USDT 托管，则：

```text
A.self_new_performance += 1000
```

### 本周团队新增业绩

用户伞下团队在本周生效的托管订单金额，不包含用户本人自己的购买金额。

例如用户 U 购买 1000 USDT，U 的所有上级都增加团队新增业绩：

```text
parent.team_new_performance += 1000
```

### 本周直推区新增业绩

某个用户下面的每个直推人，代表该用户的一条直推区。

某条直推区的本周新增业绩不是只看直推人的团队业绩，而是：

```text
直推区新增业绩 = 直推人本周个人新增业绩 + 直推人本周团队新增业绩
```

原因是直推人自己购买，也属于这条直推区的新增业绩。

### 本周新增小区业绩

用户所有直推区新增业绩汇总后，扣除最大区：

```text
本周新增小区业绩 = 所有直推区新增业绩合计 - 最大直推区新增业绩
```

如果用户本周只有 0 条或 1 条有新增业绩的直推区，则本周新增小区业绩为 0。

## 计算示例

关系如下：

```text
A
├─ B
│  └─ C
│     └─ U 入金 1000
└─ D
   └─ X 入金 3000
```

U 入金 1000 后：

```text
U.self_new_performance += 1000

C.team_new_performance += 1000
B.team_new_performance += 1000
A.team_new_performance += 1000
```

此时 A 的直推区：

```text
B 区 = B.self_new_performance + B.team_new_performance = 0 + 1000
D 区 = 0

A.community_new_performance = 1000 - 1000 = 0
```

X 入金 3000 后：

```text
X.self_new_performance += 3000

D.team_new_performance += 3000
A.team_new_performance += 3000
```

此时 A 的直推区：

```text
B 区 = 1000
D 区 = 3000

A.total_line_performance = 4000
A.max_line_performance = 3000
A.community_new_performance = 4000 - 3000 = 1000
```

## 建议表结构

### 每周新增业绩汇总表

建议新增：

```text
t_stake_hosting_weekly_community_performance
```

该表按“用户 + 周期”保存一条汇总记录。

建议字段：

```text
id                         主键
user_id                    用户ID
account                    用户钱包地址快照
week_start_time            周开始时间，格式 yyyyMMddHHmmss
week_end_time              周结束时间，格式 yyyyMMddHHmmss
self_new_performance       本周个人新增业绩
team_new_performance       本周团队新增业绩
total_line_performance     本周所有直推区新增业绩合计
max_line_performance       本周最大直推区新增业绩
community_new_performance  本周新增小区业绩
settle_status              状态 0:统计中 1:已锁定 2:已参与分红
batch_no                   分红批次号
create_time                创建时间
update_time                更新时间
remark                     备注
deleted                    删除标志
```

唯一约束：

```text
uk_user_week(user_id, week_start_time)
```

## 数据初始化与更新流程

订单生效后，需要把该订单金额写入本周新增业绩统计。

建议使用订单事件顺序消费，不建议在多个并发线程里同时直接汇总同一批用户，否则容易出现同一用户同一周累加覆盖问题。

## 本批实施计划

本批实现本周新增业绩记录能力，并将全球分红 102 权重从现有小区业绩切换为本周新增小区业绩。

实施内容：

```text
1. 托管订单表增加业绩开始时间、业绩结束时间、周新增业绩处理状态、跳过原因和处理完成时间。
2. 新增用户每周新增业绩汇总表 t_stake_hosting_weekly_community_performance。
3. 订单支付成功或后台拨付生效时，计算 performance_start_time 和 performance_end_time。
4. 如果 performance_end_time <= 本周 week_end_time，说明订单本周内到期，直接标记为已完成并记录跳过原因，不进入 Redis。
5. 如果 performance_end_time > 本周 week_end_time，说明本周结束时仍有效，标记为队列中，并发送 Redis Stream 慢慢消费。
6. Redis 消费端顺序处理订单，买家累加个人新增业绩，上级累加团队新增业绩。
7. 每次订单处理后，重算买家和所有上级的本周新增小区业绩。
8. 102 全球分红按 community_new_performance 作为同等级内的加权分配权重。
```

Redis Stream 复用现有订单动态处理队列，新增业务类型：

```text
bizType = 5：托管订单周新增业绩统计
```

这样不新增队列基础设施，只在现有消费入口中增加周新增业绩分支。

本批 SQL 必须使用独立增量脚本，不修改历史建表 SQL。

### 1. 订单进入待统计状态

托管订单支付成功或后台拨付生效后，记录订单生效时间，并标记该订单尚未完成周新增业绩统计。

建议托管订单增加字段：

```text
performance_start_time         业绩开始时间，格式 yyyyMMddHHmmss
performance_end_time           业绩结束时间，格式 yyyyMMddHHmmss
weekly_performance_status      周新增业绩统计状态 0:待处理 1:队列中 2:处理中 3:已完成 4:失败
weekly_performance_skip_reason 周新增业绩跳过原因
weekly_performance_time        周新增业绩处理完成时间
```

`performance_start_time` 用于判断订单归属到哪一周。

`performance_end_time` 根据托管订单开始时间和套餐天数计算，用于判断该订单在本周结束时是否仍然有效。

`weekly_performance_status = 3` 表示该订单的周新增业绩事件已经处理完成。这里的完成包括两种情况：

```text
1. 有效订单已经写入周新增业绩汇总。
2. 无效订单已经判断为不参与统计，并记录跳过原因。
```

## 订单回调事件处理

接收到质押托管订单回调事件后，先在订单表记录业绩开始时间、业绩结束时间和待处理状态。

建议流程：

```text
1. 订单支付或生效回调进入。
2. 计算 performance_start_time。
3. 根据 performance_start_time 和托管套餐天数计算 performance_end_time。
4. 根据 performance_start_time 计算本周 week_start_time 和 week_end_time。
5. 判断 performance_end_time 是否大于 week_end_time。
6. 如果 performance_end_time <= week_end_time，说明订单在本周内到期，属于无效新增业绩，直接标记 weekly_performance_status = 3，并记录 skip_reason。
7. 如果 performance_end_time > week_end_time，说明订单在本周结束时仍有效，标记 weekly_performance_status = 1，并发送到 Redis 队列等待顺序消费。
```

这样可以避免无效订单进入后续周新增业绩统计流程，也不会初始化购买用户和上级用户的周新增业绩数据。

示例：

```text
本周周期：2026-05-04 00:00:00 至 2026-05-10 23:59:59

订单1：
performance_start_time = 2026-05-05 10:00:00
performance_end_time   = 2026-06-04 10:00:00
判断结果：performance_end_time > week_end_time
处理方式：发送到 Redis 队列，后续顺序消费并计算小区业绩。

订单2：
performance_start_time = 2026-05-06 10:00:00
performance_end_time   = 2026-05-09 10:00:00
判断结果：performance_end_time <= week_end_time
处理方式：不发送 Redis 队列，直接标记周新增业绩处理完成，跳过原因为本周内到期。
```

### 2. 顺序消费订单事件

Redis 消费端只处理已经通过有效性判断的订单。

消费时按订单业绩开始时间顺序处理。

建议同一个消费组内保证单线程或按固定分片顺序处理，至少要保证同一个用户链路相关的汇总更新不会并发覆盖。

基础步骤：

```text
1. 查询托管订单。
2. 判断 weekly_performance_status 是否已完成，已完成则直接跳过。
3. 判断 weekly_performance_status 是否为队列中或处理中，不符合则跳过。
4. 根据 performance_start_time 计算 week_start_time 和 week_end_time。
5. 再次校验 performance_end_time 是否大于 week_end_time；如果不大于，则标记为已完成并记录跳过原因，不初始化周新增业绩。
6. 初始化或更新购买用户本周汇总记录。
7. 查询购买用户所有上级。
8. 初始化或更新每个上级本周汇总记录。
9. 重算受影响用户的本周新增小区业绩。
10. 标记订单 weekly_performance_status = 已完成。
```

### 3. 初始化或更新购买用户

如果购买用户本周记录不存在，则插入：

```text
self_new_performance = 订单金额
team_new_performance = 0
total_line_performance = 0
max_line_performance = 0
community_new_performance = 0
settle_status = 0
```

如果购买用户本周记录已存在，则更新：

```text
self_new_performance += 订单金额
```

购买用户自己的订单只增加自己的个人新增业绩，不增加自己的团队新增业绩。

### 4. 初始化或更新上级用户

通过 `t_user_relation` 查询购买用户所有上级：

```sql
select par_user_id, distance
from t_user_relation
where pos_user_id = #{buyerUserId}
  and distance > 0
  and active_flag = 1
order by distance asc;
```

对每一个上级用户：

如果该上级本周记录不存在，则插入：

```text
self_new_performance = 0
team_new_performance = 订单金额
total_line_performance = 0
max_line_performance = 0
community_new_performance = 0
settle_status = 0
```

如果该上级本周记录已存在，则更新：

```text
team_new_performance += 订单金额
```

### 5. 重算新增小区业绩

每次订单统计完成后，需要重算受影响用户的本周新增小区业绩。

受影响用户包括：

```text
购买用户本人
购买用户所有上级
```

重算某个用户时，先查询该用户所有直推人：

```sql
select user_id
from t_user_info
where invite_user_id = #{userId}
```

再读取这些直推人在本周的汇总记录：

```text
直推区新增业绩 = self_new_performance + team_new_performance
```

然后计算：

```text
total_line_performance = 所有直推区新增业绩合计
max_line_performance = 最大直推区新增业绩
community_new_performance = total_line_performance - max_line_performance
```

如果没有直推区，或者只有一条直推区有新增业绩，则：

```text
community_new_performance = 0
```

## 顺序与幂等要求

### 顺序要求

周新增业绩统计是累加型数据，必须避免并发覆盖。

建议：

- 订单事件进入 Redis Stream 或 MQ。
- 消费端按顺序处理。
- 处理单笔订单时，先更新购买用户，再更新上级团队，再重算小区。
- 同一订单只能被成功统计一次。

### 幂等要求

订单表需要记录周新增业绩统计状态。

消费前先判断：

```text
weekly_performance_status = 3
```

如果已完成，则直接跳过。

消费成功后再更新为已完成。

如果消费失败，则保留失败状态和错误信息，方便后续补偿重试。

### 初始化要求

任何用户本周记录不存在时，都要先初始化再更新。

初始化时不要依赖用户是否已经产生过团队业绩，只要本次订单影响到该用户，就需要保证该用户本周有一条汇总记录。

## 与每周全球分红的关系

每周全球分红不再使用用户当前 `community_performance` 作为权重，而是使用：

```text
t_stake_hosting_weekly_community_performance.community_new_performance
```

分红任务执行前，需要确保本周所有已生效订单的周新增业绩统计已完成。

分红发放完成后，需要把实际参与分红的周新增业绩记录更新为：

```text
settle_status = 2
batch_no = 本次全球分红批次号
```

建议 102 每周任务开始前做一次补偿扫描：

```text
查询本周已生效、weekly_performance_status != 已完成 的托管订单
按 performance_start_time 顺序补处理
```

补偿完成后，再锁定本周汇总数据并执行全球分红。

## 风险点

- 如果订单统计并发执行，同一个用户同一周的累计金额可能被覆盖。
- 如果只计算直推人的团队新增业绩，会漏掉直推人自己的新增订单。
- 如果只在每周结算时临时全量计算，App 或后台无法实时查看本周新增业绩。
- 如果用户关系后续允许变更，需要确认周新增业绩按订单生效时关系计算，还是按结算时最新关系计算。
- 如果历史订单没有 `performance_start_time` 和 `performance_end_time`，首次上线需要决定是否补历史周数据。

## 待确认问题

- 后台拨付托管订单是否计入周新增业绩。
- 周归属时间使用支付成功时间、产出开始时间，还是订单生效时间；当前建议使用订单生效时间。
- 用户关系如果发生变更，历史已统计的周新增业绩是否保持不变。
- 失败订单补偿重试是否需要后台手动入口。

## 验收用例

- 用户本人购买后，本周个人新增业绩增加，团队新增业绩不增加。
- 用户购买后，所有上级本周团队新增业绩增加。
- 直推人本人购买时，该金额计入上级对应直推区。
- 某用户只有一条直推区有新增业绩时，本周新增小区业绩为 0。
- 某用户两条直推区分别新增 1000 和 3000 时，本周新增小区业绩为 1000。
- 同一订单重复消费时，不重复增加周新增业绩。
- 用户本周记录不存在时自动插入，存在时累加更新。
- 102 每周全球分红前，未完成统计的订单可以被补偿处理。
