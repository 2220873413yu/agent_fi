# 新增小区业绩定义

## 目标

本文只定义“新增小区业绩”的业务口径，后续代码、SQL 和全球分红权重都按本文执行。

新增小区业绩不是简单把本周新增订单金额相加，而是按托管订单积分、直推区有效积分快照、大区/小区关系计算出来的周变化值。

## 基础定义

### 托管积分

托管订单参与新增小区业绩时，不直接使用 USDT 金额，而是使用“托管积分”。

```text
托管积分 = 托管金额 × 套餐积分系数
```

套餐积分系数需要在套餐配置中维护，并在用户质押托管时写入订单快照。后续套餐系数调整，不影响历史订单。

当前全球分红套餐权重：

```text
1天   = ×0
30天  = ×1
90天  = ×2
180天 = ×3
360天 = ×6
```

### 有效积分

一笔托管订单在某个统计时点仍未到期，才算有效积分。

```text
订单有效条件：performance_start_time <= 统计时点 < performance_end_time
```

如果订单已经到期，则该订单积分需要从对应线路的有效积分中扣除。

### 直推区

某个用户 A 的每个直推用户，都是 A 的一条直推区。

直推区积分不是只看直推用户本人，而是看该直推用户整条伞下当前有效托管积分：

```text
某直推区有效积分 = 直推用户本人有效托管积分 + 直推用户伞下团队有效托管积分
```

推荐关系来源：

```text
t_user_info.invite_user_id     识别直接推荐人
t_user_relation                识别祖先/后代关系
```

### 大区和小区

对用户 A 来说，当前有效积分最高的直推区是大区，其余直推区合计为小区。

```text
当前小区有效积分 = 所有直推区当前有效积分合计 - 最大直推区当前有效积分
```

如果只有 0 条或 1 条有有效积分的直推区，则当前小区有效积分为 0。

## 本周新增小区业绩

本周新增小区业绩按自然周统计：

```text
周一 00:00:00 到 周日 23:59:59
```

时间字段使用 `yyyyMMddHHmmss` 格式的 long 值。

```text
week_start_time = 20260504000000
week_end_time   = 20260510235959
```

本周新增小区业绩的核心公式：

```text
本周新增小区业绩 = 本周末小区有效积分快照 - 上周末小区有效积分快照
```

因此，本周新增小区业绩允许为负数。

### 为什么允许负数

如果本周小区内到期扣除 1000 积分，同时小区内新增 200 积分，则：

```text
本周新增小区业绩 = 200 - 1000 = -800
```

这表示该用户本周小区有效积分净减少 800。

## 自己下单与上级计算

用户自己下单，不计入自己的小区新增业绩。

原因是小区业绩只从自己的直推区中产生，自己本人不属于自己的任何直推区。

但是，用户下单会计入所有上级的对应直推区。

例如 U 下单，链路为：

```text
A
└── B
    └── C
        └── U
```

则：

```text
U 自己的小区业绩：不因为 U 自己下单而增加
C 的某条直推区：增加 U 的托管积分
B 的某条直推区：增加 U 的托管积分
A 的某条直推区：增加 U 的托管积分
```

也就是说，下单人的所有上级才会根据直推区变化重新计算小区业绩。

## 例子一：第一周初始建仓

假设：

```text
用户 A 等级：F1
F1 分红占比：30%
全网周分红池：10,000 USDT
```

A 的小区本周新增托管如下：

| 下线 | 托管金额 | 锁仓期 | 积分系数 | 产生积分 |
| --- | ---: | ---: | ---: | ---: |
| 甲 | 1,000 U | 30天 | ×1 | 1,000 |
| 乙 | 500 U | 90天 | ×2 | 1,000 |
| 丙 | 200 U | 180天 | ×3 | 600 |
| 丁 | 100 U | 360天 | ×6 | 600 |

如果这些下线都属于 A 的小区线路，且第一周没有到期扣除：

```text
A 本周小区新增积分 = 1,000 + 1,000 + 600 + 600 = 3,200
```

假设全网 F1 等级本周小区新增总积分为 50,000：

```text
A 占比 = 3,200 / 50,000 = 6.4%
F1 可分配池 = 10,000 × 30% = 3,000 USDT
A 分红 = 3,000 × 6.4% = 192 USDT
```

## 例子二：第五周持续拉新并扣除到期积分

A 的小区本周新增托管如下：

| 下线 | 金额 | 期限 | 积分系数 | 积分 |
| --- | ---: | ---: | ---: | ---: |
| 戊 | 800 U | 90天 | ×2 | 1,600 |
| 己 | 300 U | 30天 | ×1 | 300 |
| 庚 | 500 U | 180天 | ×3 | 1,500 |

本周新增积分：

```text
1,600 + 300 + 1,500 = 3,400
```

同时，第一周 30 天的 甲 订单到期：

```text
到期扣除积分 = 1,000 × 1 = 1,000
```

如果这笔到期积分属于 A 的小区线路，则：

```text
A 本周新增小区业绩 = 3,400 - 1,000 = 2,400
```

假设全网 F1 等级本周小区新增总积分为 48,000：

```text
A 占比 = 2,400 / 48,000 = 5%
F1 可分配池 = 10,000 × 30% = 3,000 USDT
A 分红 = 3,000 × 5% = 150 USDT
```

## 例子三：到期发生在大区且大区仍然不变

假设 A 有三条直推区：

```text
甲区：5,000
B区：2,000
C区：1,000
```

此时：

```text
大区 = 甲区 5,000
A 当前小区有效积分 = B区 2,000 + C区 1,000 = 3,000
```

过了 5 周后，甲区里面有 1,200 积分到期：

```text
甲区剩余 = 5,000 - 1,200 = 3,800
B区 = 2,000
C区 = 1,000
```

甲区仍然是最大区：

```text
大区 = 甲区 3,800
A 当前小区有效积分 = B区 2,000 + C区 1,000 = 3,000
```

因此这次甲区到期不会扣除 A 的小区业绩：

```text
A 本周新增小区业绩变化 = 3,000 - 3,000 = 0
```

这说明不能简单按“哪里到期就给所有上级扣小区业绩”。必须先重算直推区有效积分快照，再判断大区是否变化，最后用小区快照差额得出本周新增小区业绩。

## 代码实现原则

1. 套餐需要维护积分系数，订单需要保存积分系数快照和托管积分快照。
2. 用户自己下单，只作为自己上级链路的直推区积分变化来源，不直接增加自己的小区新增业绩。
3. 新增和到期都不能直接粗暴加减小区业绩，必须重算受影响上级的直推区有效积分快照。
4. 本周新增小区业绩等于本周末小区有效积分快照减上周末小区有效积分快照，允许出现负数。
5. 周业绩记录层必须保留真实数值，允许负数和 0，不能强制归零。
6. 全球分红发放只使用 `community_new_performance > 0` 的用户；`community_new_performance <= 0` 的用户只保留记录，不参与同等级分母，也不生成分红明细。

## 全球分红使用口径

新增小区业绩用于全球分红时，只按正数参与分配：

```text
community_new_performance > 0   参与同等级分红权重
community_new_performance = 0   不参与分红，不生成分红明细
community_new_performance < 0   不参与分红，不生成分红明细
```

负数只表示本周小区有效积分净减少，不作为扣款，也不作为负向权重抵扣同等级总分母。

等级奖池先按 `t_user_level_config.global_fee_dividend_ratio` 从全球分红总池中切分：

```text
等级奖池 = 全球分红总池 × global_fee_dividend_ratio / 100
```

同等级用户之间，再按正数周新增小区积分占比分配：

```text
用户分红 = 等级奖池 × 用户正数周新增小区积分 / 同等级正数周新增小区积分合计
```

例如同一等级内：

```text
A 本周新增小区业绩 = -800
B 本周新增小区业绩 = 0
C 本周新增小区业绩 = 4,700
D 本周新增小区业绩 = 3,000
```

则全球分红同等级总权重为：

```text
4,700 + 3,000 = 7,700
```

A 和 B 只保留周业绩记录，不生成全球分红明细。
