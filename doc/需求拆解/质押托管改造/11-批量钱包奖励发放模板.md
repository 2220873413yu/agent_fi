# 批量钱包奖励发放模板

## 适用场景

凡是定时任务或批量结算中存在“给多个用户批量增加钱包余额，同时写奖励记录”的逻辑，都优先参考 `AsyncTaskServiceImpl.distributePtbInterest100` / `distributeStakeStaticReward` 的写法。

典型场景：

- 101 托管静态收益批量发放。
- 托管直推奖、极差奖、平级奖批量发放。
- 每周全球分红批量发放。
- 其他只增加用户钱包余额的奖励。

不适用场景：

- 扣减钱包余额，例如 AFI 质押扣减。
- 需要逐笔强校验余额不能为负的扣款。
- 每笔奖励依赖上一笔落库结果才能继续计算的强串行业务。

## 核心口径

- 批量增加钱包余额时，优先构造 `List<UserMoney>`，再使用批量更新钱包方法。
- 奖励记录优先构造 `List<RewardRecord>`，再使用 `rewardRecordService.saveBatch(...)`。
- 每条 `UserMoney` 必须设置：
  - `id`：用户 ID。
  - 对应币种字段，例如 USDT 使用 `validNum1`，AFI 使用 `validNum2`。
  - `gtId`：本次钱包变动唯一流水标识。
  - `sourceCode`：来源订单号、批次号或结算单号。
  - `sourceType`：钱包流水业务类型。
  - `sourceId`：来源用户 ID 或业务来源 ID。
  - `updateTime`：本次结算时间。
- 每条 `RewardRecord` 必须设置：
  - `orderCode`：奖励记录单号。
  - `userId`：收益用户。
  - `amount`：奖励金额。
  - `coinType`：币种。
  - `sourceType`：奖励记录业务类型。
  - `sourceOrderCode`：来源订单号、批次号或结算单号。
  - `sourceUserId`：来源用户 ID。
  - `gtId`：如当前业务需要与钱包流水关联，必须设置。
  - `createTime`：本次结算时间。

## 批量大小

默认批量大小使用 `1000`。

原因：

- 避免单次事务中列表过大。
- 避免单次 SQL 太长。
- 降低定时任务内存占用。
- 和当前 `AsyncTaskServiceImpl` 里的批量发放风格保持一致。

## 推荐流程

1. 查询待处理订单、用户或分红明细。
2. 在内存中逐条计算本次应发金额。
3. 金额小于等于 `0` 的记录直接跳过。
4. 构造钱包变动 `UserMoney`。
5. 构造奖励记录 `RewardRecord`。
6. 如有订单、结算明细、批次明细，也同步加入对应待批量保存/更新列表。
7. 列表达到 `batchSize=1000` 时立即 flush。
8. 循环结束后 flush 剩余数据。
9. 任意批量更新返回失败时，必须抛异常或标记事务回滚。

## 模板代码

```java
/**
 * 批量发放某类奖励。
 *
 * <p>该方法只适用于批量增加钱包余额的奖励场景。钱包变动和奖励记录按1000条一批落库，
 * 任意批量更新失败都抛出异常并回滚当前事务。</p>
 *
 * @param rewardItems 已计算好或待计算的奖励来源数据
 * @param now 本次结算时间
 */
private void batchGrantReward(List<RewardItem> rewardItems, Date now) {
    if (CollectionUtil.isEmpty(rewardItems)) {
        return;
    }
    int batchSize = 1000;
    List<UserMoney> userMoneyList = new ArrayList<>(Math.min(rewardItems.size(), batchSize));
    List<RewardRecord> rewardRecordList = new ArrayList<>(Math.min(rewardItems.size(), batchSize));

    for (RewardItem item : rewardItems) {
        BigDecimal rewardAmount = calculateRewardAmount(item);
        if (rewardAmount == null || rewardAmount.compareTo(BigDecimal.ZERO) <= 0) {
            continue;
        }

        // 构造钱包批量增加记录，USDT奖励写validNum1。
        UserMoney money = new UserMoney();
        money.setId(item.getReceiveUserId());
        money.setValidNum1(rewardAmount);
        money.setGtId(IDUtils.getSnowflakeStr());
        money.setSourceCode(item.getSourceOrderCode());
        money.setSourceId(item.getSourceUserId());
        money.setSourceType(ConstantType.user_money_log_source_type.type_xx);
        money.setUpdateTime(now);
        userMoneyList.add(money);

        // 构造奖励记录，方便App收益记录和后台追溯。
        RewardRecord rewardRecord = new RewardRecord();
        rewardRecord.setOrderCode(IDUtils.getSnowflakeStr());
        rewardRecord.setUserId(item.getReceiveUserId());
        rewardRecord.setAmount(rewardAmount);
        rewardRecord.setCoinType(ConstantType.user_money_coin_type.type_1);
        rewardRecord.setSourceType(ConstantType.xms_reward_record_source_type.type_xx);
        rewardRecord.setSourceOrderCode(item.getSourceOrderCode());
        rewardRecord.setSourceUserId(item.getSourceUserId());
        rewardRecord.setGtId(money.getGtId());
        rewardRecord.setCreateTime(now);
        rewardRecordList.add(rewardRecord);

        if (userMoneyList.size() >= batchSize) {
            flushRewardBatch(userMoneyList, rewardRecordList);
        }
    }

    flushRewardBatch(userMoneyList, rewardRecordList);
}

/**
 * 批量落库钱包变动和奖励记录。
 *
 * @param userMoneyList 待批量增加的钱包记录
 * @param rewardRecordList 待批量保存的奖励记录
 */
private void flushRewardBatch(List<UserMoney> userMoneyList, List<RewardRecord> rewardRecordList) {
    if (CollectionUtil.isNotEmpty(userMoneyList)) {
        // USDT奖励使用validNum1批量增加；如果是AFI奖励，则改用validNum2对应的批量方法。
        bachUpdateMoneyValid1(userMoneyList);
        userMoneyList.clear();
    }
    if (CollectionUtil.isNotEmpty(rewardRecordList)) {
        rewardRecordService.saveBatch(rewardRecordList);
        rewardRecordList.clear();
    }
}
```

## 币种字段对应

- USDT：`valid_num1`，代码中使用 `validNum1`，批量方法参考 `bachUpdateMoneyValid1`。
- AFI：`valid_num2`，代码中使用 `validNum2`，批量方法参考 `bachUpdateMoneyValid2`。
- 其他币种按项目已有 `valid_numX` 口径选择对应批量方法。

## 101 托管静态收益改造提示

当前托管静态收益发放如果逐单调用 `userWalletService.handerUserMoney(...)`，功能可以跑通，但定时任务批量性能不如模板方案。

后续改造 101 时建议：

- 单笔订单先计算出静态收益、服务费、团队奖励、结算明细等结果。
- 静态收益钱包入账收集到 `List<UserMoney>`。
- 静态收益 `RewardRecord` 收集到 `List<RewardRecord>`。
- 团队奖励也按直推、极差、平级分别收集钱包记录和奖励记录。
- 订单更新、结算明细、团队收益汇总也尽量批量保存/更新。
- 最后统一 flush，失败即回滚。

## 注意事项

- 批量钱包方法适合“增加余额”，不要直接拿来做扣款。
- 批量钱包记录必须保留 `sourceCode/sourceType/sourceId/gtId`，否则钱包流水追溯会变差。
- 如果奖励记录需要和钱包流水一一对应，`RewardRecord.gtId` 建议使用同一个 `UserMoney.gtId`。
- 如果存在“同一个用户同一批次多笔奖励”，可以逐笔写钱包更新记录，保留每笔来源；不要为了合并金额丢失来源订单。
- 所有复杂批量方法必须补充方法注释和关键步骤注释。
