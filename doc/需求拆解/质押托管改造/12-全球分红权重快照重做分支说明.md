# 全球分红权重快照重做分支说明

## 当前分支状态

- 当前 `main` 分支仍是旧版全球分红/每周新增小区业绩逻辑。
- `test_01` 分支已经改过全球分红周业绩相关逻辑，但还没有完成测试。
- 后续如果要看“全球分红权重快照重做”的实现或继续测试，应先切回 `test_01` 分支；不要用 `main` 当前代码判断新逻辑是否已经落地。

## 新需求口径

- 旧的“每周新增小区业绩”逻辑整体废弃，改为“当前小区权重的周快照差值”。
- 全球分红奖池规则不变，仍按 F 等级配置 `t_user_level_config.global_fee_dividend_ratio` 切分等级奖池。
- 下单生效和订单到期先实时维护 `t_user_info` 上的当前全球分红权重，作为每周分红快照的前置条件。
- 每周全球分红任务执行时读取当前小区权重并固化周快照，再与上一期快照做差。
- 到期订单需要扣减当前权重字段，但不能修改历史周快照；历史快照只由每周 102 分红任务生成。
- 用户自己的订单不计入自己的小区权重，只计入其上级链路中的某一条直推区。

## 核心字段

### 当前权重字段

`t_user_info` 建议新增并维护以下字段，字段含义类似现有 `performance`、`umbrella_performance`、`community_performance`，但只服务于托管全球分红权重：

- `global_dividend_weight`：我的全球分红权重。
- `global_dividend_umbrella_weight`：团队全球分红权重。
- `global_dividend_community_weight`：小区全球分红权重。

当前权重维护规则：

```text
订单权重 = performance_points
        或 stake_usdt_amount × performance_coefficient 兜底

下单生效：
1. 给下单人增加 global_dividend_weight。
2. 给所有上级增加 global_dividend_umbrella_weight。
3. 重算受影响上级的 global_dividend_community_weight。

订单到期：
1. 给下单人扣减 global_dividend_weight。
2. 给所有上级扣减 global_dividend_umbrella_weight。
3. 重算受影响上级的 global_dividend_community_weight。

直推区权重 = 直推用户 global_dividend_weight + global_dividend_umbrella_weight
小区权重 = 所有直推区权重合计 - 最大直推区权重
```

### 周快照字段

权重快照表建议使用 `t_stake_hosting_global_dividend_weight_snapshot`，替代旧的 `t_stake_hosting_weekly_community_performance`。

- `community_weight`：本周 102 分红任务执行时读取到的 `t_user_info.global_dividend_community_weight`。
- `previous_community_weight`：上一周结算快照里的 `community_weight`。
- `dividend_weight`：本周参与分红权重，等于 `max(community_weight - previous_community_weight, 0)`。
- `self_weight`：本周快照时的 `global_dividend_weight`，用于排查。
- `umbrella_weight`：本周快照时的 `global_dividend_umbrella_weight`，用于排查。
- `settle_status`、`batch_no`：用于标记该快照是否已参与全球分红。

## 结算流程

1. 102 每周全球分红任务定位本次自然周周期。
2. 查询用户当前全球分红权重字段，读取 `global_dividend_weight`、`global_dividend_umbrella_weight`、`global_dividend_community_weight`。
3. 查询该用户上一期权重快照，取上一期 `community_weight` 作为 `previous_community_weight`，没有上一期则为 `0`。
4. 写入本期权重快照：`community_weight = global_dividend_community_weight`。
5. 计算 `dividend_weight = max(community_weight - previous_community_weight, 0)`。
6. 所有用户都必须保存本期快照，即使 `community_weight` 下降、为 0，或 `dividend_weight = 0`。
7. 只取 `dividend_weight > 0` 且有效用户参与分红。
8. 按 F 等级分池后，在同等级内按 `dividend_weight / level_dividend_weight` 分配。

必须保存下降用户的快照。例如 A 第一周 `community_weight=10000`，第二周降到 `8000` 时也要保存第二周快照；第三周升到 `12000` 时应按 `12000 - 8000 = 4000` 参与分红，而不是按 `12000 - 10000`。

## 旧链路停用策略

- 旧 `t_stake_hosting_weekly_community_performance` 收集链路本次先停用发送和消费入口。
- 旧 service、mapper、表、订单 `weekly_performance_*` / `weekly_expire_performance_*` 字段暂时保留，不立即删除。
- `bizType=5`、`bizType=7` 不再触发旧周新增/周到期小区业绩收集。
- `bizType=6` 继续保留 G7、新托管小区业绩和等级重算，只跳过旧周新增收集。
- 后续确认没有依赖后，再单独删除旧结构和后台旧菜单。

## SQL 与表结构

本次先不删除旧结构，只停用旧收集入口。后续确认没有依赖后，再单独执行旧结构清理：

```sql
DROP TABLE IF EXISTS t_stake_hosting_weekly_community_performance;
DROP TABLE IF EXISTS t_stake_hosting_global_dividend_detail;
DROP TABLE IF EXISTS t_stake_hosting_global_dividend_batch;
```

后续实现新口径时需要新建或调整：

- `t_user_info`：新增当前全球分红权重字段。
- `t_stake_hosting_global_dividend_weight_snapshot`：全球分红权重快照表。
- `t_stake_hosting_global_dividend_batch`：全球分红批次表。
- `t_stake_hosting_global_dividend_detail`：全球分红明细表。
- 批次状态字典。
- 权重快照结算状态字典。
- 后台菜单由旧“每周新增小区业绩”调整为“全球分红权重快照”，权限前缀建议使用 `xms:stakeHostingGlobalDividendWeightSnapshot:*`。

## 测试提示

测试前先确认当前分支：

```bash
git branch --show-current
```

应切到 `test_01` 后再测试新逻辑：

```bash
git checkout test_01
```

重点验证场景：

- 第一周：A=10000、B=5000、C=3000，三人都参与分红。
- 第二周：A 当前小区权重低于上期时不参与；B 新增 1000；C 新增 100。
- 第三周：A 新增 4000；B 无新增不参与；C 新增 1900。
- 权重下降时，`dividend_weight = 0`，不生成分红明细。
- 不同 F 等级只在各自等级奖池内分配。
- 无参与用户时，批次生成但实际发放为 0，奖池处理沿用现有空跑或失败规则。
